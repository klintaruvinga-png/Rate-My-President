## 1. EOM Automation Contract

The Executive Operations Manager (EOM) is the cross-project coordination layer. It runs on
Hermes primitives (skills + crons), not a separate service.

### 1.1 Registry is the source of truth
- `EOM/projects.json` is the single registry of projects, tasks, milestones, and blockers.
- Per-project `TRACKER.md` files are **derived** — generated by `EOM/sync_trackers.py`.
- Rule: update `projects.json`, then run `sync_trackers.py`. Never hand-edit a TRACKER as the
  source of truth (it will be overwritten on next sync).
- Task status vocabulary: `not_started | in_progress | partial | done | blocked | missing`.

### 1.2 Cron cadence (Hermes)
- Daily Standup — 08:00, skill `eom`, delivers to origin + Telegram.
- End-of-Day Review — 21:00, skill `eom`, delivers to origin + Telegram.
- Weekly Review — Mon 09:00, skill `eom`.
- Self-Improvement — Sun 22:00, skills `eom` + `eom-self-improve`.
- BrainWorks Curator — 07:00 daily (OpenClaw curation pass).
- All cron jobs run on the **global default model** (`tencent/hy3:free` via `nous`). Do NOT pin
  them to a specific OpenRouter free model — those hit daily quota 429s and fail silently.
  If a cron fails with "Rate limit exceeded: free-models-per-day", repin to the default.

### 1.3 PR / git rules (apply to EVERY project repo)
- Feature branches only. Never commit directly to `main`/`master`.
- Open **real (non-draft)** PRs: `gh pr create` without `--draft`. Drafts skip auto-review.
- PR body must cite exact validation commands + results (build output, test counts).
- Do not force push. Do not delete branches. Do not remove lockfiles without cause.
- Before risky git moves, preserve loose work to a `wip/...` branch.
- When a session ends with file changes, it MUST end on a feature branch with an open PR.
  If read-only, state that no branch/PR was created.

### 1.4 Verification rule (Kudzie's hard rule)
- **Verify, don't assume.** Run the project's build/typecheck/test before claiming done.
- Never report "passes" without real tool output. If validation is unavailable, say so.
- For EOM-tracked projects, the acceptance gate is encoded in `projects.json`
  (`acceptance_gate.build_required`, `verify_required`, `must_pass_ci`).

### 1.5 BrainWorks append pipeline
- The EOM End-of-Day cron appends a session evidence JSONL entry to BrainWorks
  (`observations.jsonl`) per the producer protocol (§2). Confirm the first real write landed.

## 2. BrainWorks Producer Protocol

BrainWorks is a cognitive digital twin, not a notes app. It lives at
`C:\Users\Kudzie\OneDrive\BrainWorks`.

- Read `BrainWorks.md` before significant planning, implementation, or decision-making work.
- Treat `BrainWorks.md` as curated memory; `observations.jsonl` is raw evidence.
- **You are a producer, not the curator. Never edit `BrainWorks.md` directly.**
- At the end of each significant chat, editor, or CLI coding session, append one canonical
  JSONL entry to `observations.jsonl`.
- Use the PowerShell helper:
  `@'...json...'@ | powershell -File C:\Users\Kudzie\OneDrive\BrainWorks\append_observation.ps1`
- Follow the exact schema in `C:\Users\Kudzie\OneDrive\BrainWorks\agent_write_instruction.md`.
- **OpenClaw is the only curator** allowed to promote evidence into `BrainWorks.md`.
- Cognitive-extension layers (log when observed, in addition to the mandatory pair):
  - `knowledge`: Kudzie's grasp (0–100) + self-confidence (0–100) of a topic; emit a metric.
  - `gap`: misunderstanding or repeatedly asked topic; emit metric with seen/importance/recommendation.
  - `mistake`: repeated error pattern across projects; emit metric with occurred/likelihood/recommendation.
  - `bias`: recurring reasoning preference or blind spot.
  - `belief`: stated opinion that changed; emit metric with prior/current/evidence.
- Security: never store passwords, credentials, raw chat logs, or temporary emotions in
  BrainWorks. `observations.jsonl` holds distilled observations, not transcript dumps.
